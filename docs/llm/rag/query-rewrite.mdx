# Query Rewrite

## 本章目标

这一章讨论 RAG 里一个非常实用、但经常被低估的优化点：`Query Rewrite`，也就是查询改写。

读完后你应该能：

- 理解为什么用户原始问题不一定适合直接拿去检索
- 知道哪些问题需要改写
- 写出一个最小查询改写 Prompt
- 理解改写和生成回答是两个不同阶段

---

## 为什么原始问题常常不适合直接检索

用户提问在真实世界里通常有几个特点：

- 太口语化
- 太短
- 省略上下文
- 混有代词和简称
- 目标不明确

例如：

```text
这个离职要提前多久？
```

这里的“这个”到底指试用期离职、正式员工离职，还是合同终止？

如果直接检索，可能召回一堆泛相关内容。

---

## Query Rewrite 的作用

Query Rewrite 的目标是：

> 把用户原始问题改写成更适合知识检索的表达。

它通常会做这些事：

- 补全主题
- 去除口语噪声
- 替换成更正式的检索表达
- 明确关键实体和条件


## 1. 一个最简单的改写例子

原问题：

```text
年假怎么结转？
```

改写后：

```text
公司人事制度中，员工未休完年假最多可结转多少天，以及结转后的使用期限是什么？
```

改写后的查询通常更容易命中：

- 年假
- 结转
- 使用期限

这些关键概念。

---

## 2. 一个教学版改写 Prompt

```python
def build_query_rewrite_prompt(question: str) -> str:
    return f"""
    你是一名检索优化助手。
    请将下面用户问题改写成更适合企业知识库检索的查询语句。

    要求：
    - 保留原问题意图
    - 补全可能缺失的关键主题
    - 使用正式、清晰、可检索的表达
    - 只输出改写后的查询，不要解释

    用户问题：{question}
    """
```

## 3. 哪些问题最适合改写

### 口语问题

例如：

- “这个怎么报销？”
- “试用期那个离职咋算？”

### 缩写太多的问题

例如：

- “v2 上传限啥格式？”

### 多轮对话里的追问

例如：

- “那这个还能延期吗？”

这类问题如果不结合上下文改写，检索质量通常很差。

---

## 4. 两个业务案例

### 案例一：制度问答

原问题：

```text
试用期那个提前多久说？
```

改写后：

```text
公司制度中，试用期员工离职是否需要提前通知，以及具体需要提前几天？
```

### 案例二：前端研发问答

原问题：

```text
白屏那个 chunk 的问题咋搞？
```

改写后：

```text
前端部署后因 chunk load error 导致页面白屏的原因和排查方法是什么？
```

---

## 5. Query Rewrite 不是越复杂越好

如果问题本身已经非常明确，例如：

```text
上传接口支持哪些文件类型？
```

再大幅改写未必有收益，甚至可能引入噪声。

所以要注意：

- 明确问题适度改写
- 模糊问题优先改写

---

## 6. 常见坑

### 坑一：改写改变了原始意图

这会导致“检索更准了，但答的不是用户真正问的内容”。

### 坑二：改写太长太复杂

可能加入无关约束，反而影响召回。

### 坑三：所有问题一律改写

会增加延迟和成本，也未必必要。

---

## 本章小结

你现在应该记住：

- 用户原问题不一定适合直接检索
- Query Rewrite 的目标是提高召回质量，不是直接回答问题
- 口语化、缩写化、上下文依赖强的问题最适合改写
- 改写必须保留用户原始意图，不能自作主张换题
