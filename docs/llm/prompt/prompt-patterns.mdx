# 常见 Prompt 模式

## 本章目标

这一章把 Prompt 设计中最常见、最有复用价值的模式系统化整理出来。

读完后你应该能：

- 把 Prompt 从“临场发挥”升级成“模式化设计”
- 根据任务类型快速选择合适模式
- 将 Prompt 模式抽象成模板和函数

---

## 为什么要学模式

如果你每次写 Prompt 都从头想，会遇到几个问题：

- 不稳定
- 不好复用
- 不好调试
- 很难沉淀经验

而模式化设计的价值就在于：

> 让 Prompt 像工程模板一样可以复用、比较和迭代。

---

## 1. 角色 + 任务 + 约束 + 输出格式

这是最基础、最通用的一类模式。

```text
你是一名{角色}。
请完成{任务}。

要求：
- {约束1}
- {约束2}

输出格式：{格式要求}
```

适合几乎所有任务，是 Prompt 工程的基础骨架。

---

## 2. 先分析再回答

适合：

- 复杂需求分析
- 排障
- 诊断型问题

例如：

```text
请先分析问题，再给结论，最后给建议。
```

这类模式的好处是让模型不要一上来就拍最终答案，而是先把推理步骤显性化。

---

## 3. 先引用依据再回答

适合：

- RAG
- 知识问答
- 制度类、文档类回答

例如：

```text
请先列出依据片段，再给最终结论。如果依据不足，请明确说明。
```

这种模式能显著提升：

- 可解释性
- 用户信任感

---

## 4. 先分类再路由

适合：

- 工单分诊
- 工具选择
- 多路径任务处理

例如：

```text
请先判断问题属于哪一类，再按对应策略输出。
```

这类模式非常适合后面的 Tool Calling 和 Agent。

---

## 5. 先自检再输出

适合：

- 高风险结构化输出
- 要求字段完整的系统任务

例如：

```text
输出前请检查：
- 是否字段完整
- 是否有编造
- 是否满足格式要求
```

这种模式并不能保证万无一失，但通常能让结果更稳一点。

---

## 6. 一个模板化 Prompt 示例

```python
from textwrap import dedent


def build_analysis_prompt(requirement: str) -> str:
    return dedent(f"""
    你是一名 AI 技术方案顾问。
    请分析下面需求。

    输出：
    1. 问题定义
    2. 输入输出
    3. 是否需要 RAG
    4. 是否需要 Tool Calling
    5. MVP 建议

    需求：
    {requirement}
    """)
```

模板化的价值在于：

- 易复用
- 易版本化
- 易做评测对比

---

## 7. 三个业务案例

### 案例一：企业知识库助手

适合模式：

- 先引用依据再回答

### 案例二：工单自动分诊

适合模式：

- 先分类再路由

### 案例三：前端错误修复建议

适合模式：

- 先分析再回答

---

## 8. 常见坑

### 坑一：模式乱叠

把很多模式生硬堆在一起，会让 Prompt 过于冗长、重点不清。

### 坑二：模式和任务不匹配

例如一个简单概念问答，没必要强行上“多层分析 + 自检 + 分类 + 结构化”。

### 坑三：不抽象模板

每次都重新写，长期很难沉淀。

---

## 本章小结

本章最重要的结论是：

- Prompt 可以像工程模板一样模式化设计
- 常见模式包括：角色任务约束输出、先分析再回答、先引用依据再回答、先分类再路由、先自检再输出
- 模式化能帮助你更稳定地复用和调优 Prompt
