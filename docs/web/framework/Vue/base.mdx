# 简介

## 为什么要研究Vue3源码

在开始实现 mini-vue 之前，我想先回答一个问题：

我们为什么要研究 Vue3？

如果只是为了“会用 Vue”，其实学会 ref、reactive、computed、watch、组件通信、路由、状态管理，基本就可以完成大部分业务开发了。

但如果想进一步理解现代前端框架的本质，就不能只停留在 API 层面。

Vue3 背后真正值得研究的，不只是某几个语法，而是它如何解决这些问题：

```plain
1. 数据变化之后，页面如何自动更新？
2. 组件为什么可以像函数一样组合？
3. 模板为什么能变成真实 DOM？
4. 虚拟 DOM 到底解决了什么？
5. 响应式系统如何知道哪些地方需要更新？
6. 框架如何在性能和可维护性之间做取舍？
7. 运行时和编译时分别承担了什么职责？
```

这些问题不是 Vue 独有的，也是 React、Solid、Svelte 等现代前端框架共同面对的问题。


所以研究 Vue3，本质上是在研究：

> 现代前端框架如何把复杂的 UI 更新过程变得可维护、可预测、可优化。





## 为什么会有声明式框架

早期js、jQuery时期要把html静态页面写完之后，还需要写js脚本。有些时候因为一个数据修改就要同时获取、修改一堆dom连锁反应，心智负担很重，很难维护。

```js
let count = 0

const btn = document.querySelector('#btn')
const text = document.querySelector('#text')

btn.addEventListener('click', () => {
  count++
  text.innerText = count
})
```

但是实际业务不可能只有一个count这么简单，这时候想如果我定义一个变量，只要我修改了这个变量，dom就自动给我update。

```js
let count = 0

count++

update(() => {
  // 所有关联count变量的dom都自动更新
})
```

只需要关系变量，不需要关系dom。这时候Vue与React就是自动update的框架。

开发者只需要关心数据，框架负责页面怎么更新。这也就从以前的声明式开发，转变为命令式开发。

```plain
命令式：数据变化 -> 你手动找 DOM -> 你手动改 DOM
声明式：数据变化 -> 框架自动更新页面
```

## 声明式性能问题

Vue帮我们自动更新页面，这也意味着框架中间做了很多事情，性能会比我写的`text.innerText = count`要高?

如果但看一次简单DOM更新，原生DOM通常更直接，开销更小。

```plain
命令式修改：text.innerText = count
声明式修改：修改响应式数据 -> 触发依赖 -> 调度更新 -> 重新执行渲染函数 -> 生成新DOM -> diff -> patch DOM
```

框架确实有额外成本，但是框架不是为了让每一次最简单的DOM操作都比手写更快，而是为了在复杂应用中，用统一的机制管理状态变化和视图更新。

当应用变复杂之后，性能问题不只是"某一次DOM操作快不快"，还包括：

```plain
是否重复更新？
是否更新了不该更新的地方？
是否多次触发布局和重绘？
是否能批量处理状态变化？
是否能跳过静态内容？
是否能把更新范围控制在组件级别？
```

框架的优化空间也正是在这里, Vue3会通过响应式系统知道：

```plain
哪些副作用依赖了这个数据
哪些组件需要重新渲染
哪些 DOM 节点需要更新
哪些内容是静态的，可以跳过
```

所以在复杂场景下，框架虽然有运行时成本，但它可以通过一整套机制降低不必要的更新。这是一种权衡：**牺牲一点底层直接性，换取复杂业务下更好的维护性和稳定的整体性能**

## 虚拟DOM性能

虚拟DOM并不一定比直接操作真实DOM更快。如果明确知道要更新哪个DOM,直接写

```js
text.innerText = count
```

一定比创建 vnode、diff、patch 更直接。虚拟 DOM 的价值不在于“永远比原生 DOM 快”，而在于：

> 它提供了一种用 JavaScript 对象描述 UI 的方式，让框架可以在更新前进行比较和优化。

虚拟 DOM 本质上就是普通 JavaScript 对象。

<Tabs>
<Tab label="真实DOM">
```html 
<div id="app">
  <span>Hello</span>
</div>
```
</Tab>
<Tab label="虚拟DOM">
```js
const vNode = {
  tag: 'div',
  props: {
    id: 'app'
  },
  children: [
    {
      tag: 'span',
      children: ['Hello']
    }
  ]
}
```
</Tab>
<Tab label="状态更新后虚拟DOM">
```js
const newVNode = {
  type: 'div',
  props: {
    id: 'app'
  },
  children: [
    {
      type: 'span',
      children: 'World'
    }
  ]
}
```
</Tab>
</Tabs>


比较新旧 vnode，找出变化的地方，最后只更新必要的真实 DOM。


这个过程大概是：

```plain
state change -> render -> new vnode -> diff old vnode and new vnode -> patch real DOM
```

所以虚拟 DOM 的价值主要有几个：

1. 用 JS 对象描述 UI，增强抽象能力
2. 让声明式 UI 成为可能
3. 通过 diff 找出需要更新的地方
4. 跨平台渲染更容易，比如浏览器、小程序、原生端
5. 给编译器和运行时提供统一的中间表示

> 当我不知道具体哪里变了时，如何通过比较前后两棵 UI 树，找到最小更新范围。

所以虚拟 DOM 是一种折中方案。它既不像原生 DOM 那样完全手动、不可控，也不像完全编译时框架那样提前确定所有变化。

它让框架可以在运行时保留足够的灵活性。


## Vue3编译时

Vue3 除了运行时之外，还有一个很重要的编译时。也就是Vue3的模板编译。

```html
<div>
  <p>hello</p>
  <p>{{ count }}</p>
</div>
```


不会直接原样交给浏览器，而是会被 Vue 编译器转换成渲染函数。大致可以理解成：


```js
function render() {
  return h('div', null, [
    h('p', null, 'hello'),
    h('p', null, count.value)
  ])
}
```

不过 Vue3 的编译器不会只是简单地把模板转成 h 函数。它还会在编译阶段分析模板结构，并打上优化标记。比如：

```html
<div>
  <p>hello</p>
  <p>{{ count }}</p>
</div>
```

编译器可以看出来：

```plain
第一个 p 是静态节点，永远不会变
第二个 p 里面有动态文本，count 变了才需要更新
```

这样运行时更新时，就不用傻傻地整棵树都比较一遍。 Vue3 会尽可能把能在编译阶段确定的信息提前确定下来。这也是 Vue3 性能优化中非常关键的一点。


## mini-vue 要实现什么？

所以，这个 mini-vue 系列并不是为了完整复刻 Vue3。

Vue3 的源码非常庞大，里面有大量边界情况、平台兼容、性能优化和工程设计。

mini-vue 更适合作为一条学习主线：

1. 实现响应式系统
2. 实现 effect 依赖收集和触发更新
3. 实现 computed
4. 实现 watch
5. 实现 vnode
6. 实现 renderer
7. 实现组件机制
8. 实现 setup
9. 实现 props / emit
10. 实现 slots
11. 实现 provide / inject
12. 实现模板编译
13. 实现 patchChildren
14. 实现 keyed diff

通过这些核心模块，我们可以逐步理解:

- Vue 如何知道数据变了？
- Vue 如何知道哪些地方要更新？
- Vue 如何把模板变成渲染函数？
- Vue 如何把 vnode 渲染成真实 DOM？
- Vue 如何更新组件？
- Vue 如何做 diff？
- Vue 如何把声明式代码变成命令式 DOM 操作？



## 结语

现代前端框架的出现，并不是因为原生 DOM 做不到。恰恰相反，框架底层最终依然要回到原生 DOM 操作。真正的区别在于：

> 原生 DOM 让开发者自己维护状态和视图的同步关系
> Vue3 让开发者描述状态和视图的关系，由框架负责同步

这种模式带来了额外的运行时成本，但也换来了更好的可维护性、可预测性和工程化能力。

Vue3 通过响应式系统解决“数据变化如何被感知”，通过虚拟 DOM 和渲染器解决“变化如何更新到页面”，通过编译器解决“哪些内容可以提前优化”。

所以研究 Vue3，不只是研究一个框架的源码，而是在研究现代前端框架最核心的问题：

> 当状态变化时，如何高效、正确、可维护地更新用户界面。
