# 混合检索

## 本章目标

这一章讲的是：为什么很多时候只靠向量检索还不够，为什么真实系统常常会把语义检索和关键词检索结合起来。

---

## 1. 两种检索各自擅长什么

### 语义检索（Dense Retrieval）

擅长：

- 语义接近但字面不同的问题
- 口语化表达
- 同义问法

### 关键词检索（Sparse Retrieval）

擅长：

- 精确术语
- 错误码
- 接口名
- 版本号
- 规则编号

这两种能力并不冲突，反而经常互补。

---

## 2. 为什么混合检索很常见

因为真实问题通常既有“语义”，也有“精确词”。

比如问题：

```text
上传接口支持 png 和 webp 吗？
```

这里既有接口语义，也有明确的文件类型关键词。

如果只靠语义检索，可能会召回“文件上传概览”；
如果只靠关键词检索，可能又漏掉更全面的接口说明。

因此混合检索往往更稳。


---

## 3. 哪些场景特别适合混合检索

- API 文档问答
- 错误码查询
- 规则编号、条款编号查询
- 带版本号、文件类型、接口路径的问题
- 技术文档与 FAQ 混合检索

---

## 4. 一个简化理解示例

你可以先不纠结底层实现，把它理解成：

- 一路从 embedding 角度找“意思像”的文档
- 一路从关键词角度找“字面命中”的文档
- 再把两边结果合并后排序

这个思路非常适合工程理解。

---

## 5. 两个业务案例

### 案例一：错误码检索

用户问：

```text
ERR_CHUNK_LOAD_FAIL 是什么问题？
```

这里关键词 `ERR_CHUNK_LOAD_FAIL` 很关键，关键词检索优势明显；但语义检索还能帮助召回“静态资源更新导致缓存失效”的解释文档。

### 案例二：产品文档问答

用户问：

```text
v2.3 版本的上传接口限制是什么？
```

这里：

- `v2.3` 是精确过滤信息
- “上传接口限制”是语义主题

混合检索通常比单一路径更稳。

---

## 6. 常见坑

### 坑一：盲目上混合检索

如果文档小、问题简单、关键词不明显，混合检索带来的复杂度未必值得。

### 坑二：合并后不做去重和排序

两路结果混在一起但没有融合策略，容易把结果搞乱。

### 坑三：忽视业务特征

不是所有业务都需要强关键词能力，但 API、错误码、版本号场景通常特别需要。

---

## 本章小结

你现在应该记住：

- 语义检索擅长找“意思接近”
- 关键词检索擅长找“字面精确命中”
- 混合检索的价值在于让两者互补
- 是否需要混合检索，要看你的数据和问题类型，而不是盲目追求复杂度
